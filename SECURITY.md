## 二、报告安全漏洞

### 2.1 报告方式

**请不要**通过公开的 GitHub Issues 报告安全问题。

请通过 [GitHub 安全公告页面](https://github.com/kscz0000/Batch-Translation-of-Skills/security/advisories) 私下报告安全问题。

### 2.2 报告内容

您的报告应包含：

- 问题类型和描述
- 复现步骤
- 可能的利用方式
- 建议的修复方案（可选）

### 2.3 响应时间

| 时间范围 | 承诺 |
|---------|------|
| 24 小时内 | 确认收到报告 |
| 72 小时内 | 提供初步评估 |
| 7 天内 | 发布修复版本（如适用）|

## 三、已知安全问题

### 3.1 API Key 管理

**问题**：如果用户在代码中硬编码 API Key，可能导致凭据泄露。

**解决方案**：
- 所有 API Key 通过环境变量配置
- 使用 `.env.example` 模板文件
- `.env` 文件已添加到 `.gitignore`

### 3.2 路径注入

**问题**：硬编码的绝对路径可能泄露用户目录结构。

**解决方案**：
- 使用相对路径或环境变量
- 不在代码中存储用户特定路径

---

## 四、安全最佳实践

### 4.1 对于用户

1. **永远不要**将 `.env` 文件提交到版本控制
2. **始终使用**环境变量存储敏感配置
3. **定期轮换** API Key
4. **限制 API Key 权限**，只授予必要的权限

### 4.2 对于开发者

1. **不硬编码**任何敏感信息
2. **使用安全模板**（如 `.env.example`）
3. **遵循提交前检查清单**
4. **使用自动化工具**检测敏感信息

## 五、依赖安全

### 5.1 依赖更新

定期检查并更新依赖版本：

```bash
pip list --outdated
pip install --upgrade <package>
```

### 5.2 安全扫描

推荐使用以下工具扫描依赖漏洞：

- `pip-audit`：扫描已知漏洞
- `safety`：检查依赖安全性

```bash
pip install pip-audit safety
pip-audit
safety check
```
